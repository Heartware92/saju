## 🔴 결함 (즉시 수정 필요)

### 1. 궁합 — 보관함 저장 누락 (전 카테고리)
- **위치**: `src/pages/GunghapPage.tsx:292~470` (handleAnalyze)
- **재현**: 어떤 궁합(연인/배우자/친구/펫 등) 분석해도 Supabase `saju_records` INSERT 안 됨. `grep "archiveSaju\|saveRecord" GunghapPage.tsx` → 0건
- **영향**: 사용자가 결제 후 받은 풀이가 영구 보존 안 됨. 보관함 진입 코드(`GunghapPage.tsx:251~273`)는 존재하지만 저장 경로가 없어 영원히 도달 불가. **결제 vs 가치 미스매치 → 신뢰 큰 손상**
- **수정안**: `handleAnalyze` 성공 직후 `archiveSaju({ category:'gunghap', sourceBirth:sourceBirthFromSaju(myResult), engineResult:{ category, otherProfile/petInfo, myRole, otherRole }, interpretation:cleaned, partner:{ name:otherDisplayName, birth_date:otherBase.birth_date ?? '' } })` 호출. pet 분기는 partner.birth_date 가 비어있어 별도 메타로

### 2. 궁합 — `injectRoleContext` no-op
- **위치**: `src/constants/prompts.ts:2839`
- **재현**: `prompt.replace('[작성 지침]', block + '[작성 지침]')` — 정확 문자열 매칭. 모든 gunghap 프롬프트는 `[작성 지침 — 아래 N개 섹션을 순서대로 작성하세요]` 형태(line 2474, 2558, 2644, 2727, 2800, 2909, 3022, 3114, 3212, 3300, 3391, 3493, 3597, 3708) 라 닫는 대괄호 앞에 ` —` 가 붙어 있다. 결과: replace 매치 실패 → `prompt` 원본 반환
- **영향**: GunghapPage `myRole`, `otherRole` 입력값이 AI 에 전혀 전달 안 됨. 프롬프트에 "두 사람의 역할" 블록 자체가 부재. **직원 피드백 6번을 해소하려 추가한 강제 룰이 실제로는 한 번도 작동하지 않음**
- **수정안**:
  - (a) `injectRoleContext` 의 검색 문자열을 `[작성 지침 —` 부분일치로 변경
  - (b) 또는 모든 gunghap 프롬프트의 `[작성 지침 — ...]` 앞에 placeholder 마커 (예: `[ROLE_INJECTION_POINT]`) 추가 후 그걸 replace

### 3. 지정일 운세 — 새 7문장 프롬프트 미적용
- **위치**: `src/app/saju/date/page.tsx:17` ↔ `src/pages/TodayFortunePage.tsx:180` ↔ `src/services/fortuneService.ts:653~673`
- **재현**: `/saju/date` 라우트는 `TodayFortunePage mode="date"` 를 마운트. 이 페이지는 `getTodayFortuneReport(result, confirmedDate)` → `generateTodayFortunePrompt`(5섹션 today_*) 를 호출. 직원 피드백 7번 "지정일 7문장 확장" 이 적용된 `generatePeriodDomainsPrompt`(prompts.ts:1100, 5영역×7문장) 는 **`PeriodFortunePage` 의 scope='day'·'date' 분기에서만 호출** 되는데, 이 분기는 어디서도 마운트 안 됨 (앱 라우트는 newyear scope='year' 만 사용)
- **영향**: 오늘 작업 중 핵심 변경 하나가 사용자에게 도달 안 함. **직원 피드백 미반영 상태**
- **수정안**:
  - (a) `/saju/date/page.tsx` 가 `PeriodFortunePage scope="date"` 를 마운트하도록 변경
  - (b) 또는 `TodayFortunePage mode="date"` 분기에서 `getPeriodDomainsDescription` 을 호출하도록 분기

## 🟡 개선 권장 (작동은 하나 잠재 위험)

### 4. 정통사주 — 프롬프트 룰 vs 실제 섹션 수 불일치
- **위치**: `src/constants/prompts.ts:965` 와 `JUNGTONGSAJU_SECTION_KEYS:809~811`
- **현상**: 본문 룰 9번이 "아래 11개 마커를 정확히 사용" 명시. 그러나 실제 KEYS 는 12개 (general·daymaster·element·interaction·character·career·wealth·love·health·relation·luck·advice). prompt 본문에도 12개 섹션이 정의됨(973~1080)
- **영향**: AI 가 룰을 곧이곧대로 따르면 11개만 출력해 1개 섹션이 누락될 가능성. 실제 응답에서 advice 또는 interaction 빠지면 SajuResultPage 에서 해당 카드 사라지는 형태로 부분 손실
- **수정안**: "12개 마커" 로 텍스트 정정

### 5. 지정일 운세 — `/saju/date?date=YYYY-MM-DD` 쿼리 무시
- **위치**: `src/pages/SajuInputPage.tsx:252~254` ↔ `src/pages/TodayFortunePage.tsx:91~95`
- **현상**: 입력 페이지에서 `?date=` 붙여 push 하지만, TodayFortunePage 는 `searchParams.get('date')` 안 읽고 무조건 오늘 ISO 시작
- **영향**: 사용자가 입력 페이지에서 고른 날짜가 사라져 결과 페이지에서 다시 골라야 함 (동선 회귀)
- **수정안**: TodayFortunePage 초기화 시 `const initial = searchParams?.get('date') || todayIso;`

### 6. 지정일 운세 — 다른 해 날짜 선택 시 currentSeWoon 어긋남
- **위치**: `src/constants/prompts.ts:716, 744` (`generateTodayFortunePrompt`)
- **현상**: 프롬프트 "세운(올해)" 라인이 `result.currentSeWoon`(SajuResult 계산 시점 = 오늘 연도) 사용. 사용자가 2027-03-15 같은 미래 날짜 picker 로 골라도 세운은 2026 데이터로 주입
- **영향**: 일진(today_lucky 등 4층 운기) 문맥에서 "세운" 정보만 어긋남. AI 가 맞지 않는 세운 십성을 근거로 풀이
- **수정안**: `isoDate.split('-')[0]` 으로 연도 뽑아 `result.seWoon.find(s => s.year === pickedYear)` 동적 선택

### 7. 토정비결·자미두수 — archiveSaju 에 sourceBirth 누락
- **위치**: `src/services/fortuneService.ts:365, 411`
- **현상**: `archiveSaju({ category:'tojeong'/'zamidusu', engineResult:..., interpretation:... })` 에 `sourceBirth` 빠져 있음. archiveService 매칭 로직(`archiveService.ts:96~110`) 은 sourceBirth 가 없으면 곧장 대표 프로필 fallback
- **영향**: 사용자가 URL 로 다른 사람 사주 넘겨 토정/자미두수 풀이 보면, 보관함 기록은 항상 대표 프로필 이름으로 잘못 저장. 정통사주·신년·오늘·택일은 sourceBirth 잘 넘기는 것과 비대칭
- **수정안**: 두 함수에 `sourceBirth: sourceBirthFromSaju(saju)` 추가

### 8. 보관함 카테고리 'period' 미사용
- **위치**: `src/services/archiveService.ts:28`
- **현상**: ArchiveCategory 에 'period' 정의되어 있으나 어느 호출자도 사용 안 함. /saju/date 결과는 모두 'today' 로 저장
- **영향**: 보관함에서 "지정일 운세" 와 "오늘의 운세" 가 한 카테고리로 섞여 표시. 사용자 분류 혼동
- **수정안**: TodayFortunePage mode='date' 분기에서 archiveSaju category='period' 로 (위 #3 과 함께 처리하면 자연 해결)

### 9. MoreFortuneConfig.maxTokens — 사용처 없음
- **위치**: `src/constants/moreFortunes.ts:40, 90, 109, 118, 127, 137`
- **현상**: 각 더많은운세 config 에 `maxTokens: 1300/1500/1800` 정의됨. 그러나 fortuneService 의 `getStudyShort`/`getDreamInterpretation` 등은 **하드코딩 maxTokens**(2000/2500) 사용. config 값은 어디서도 read 안 됨
- **영향**: 분량 조정이 두 군데(prompts.ts + fortuneService 호출 + config) 산재. 운영자가 config 만 보고 토큰 늘려도 무효
- **수정안**: fortuneService `getXxxShort` 들이 config 에서 maxTokens import. 또는 config 의 maxTokens 필드 제거

### 10. 자미두수 — hourUnknown 가드가 보관함 재생까지 막음
- **위치**: `src/pages/ZamidusuResultPage.tsx:229~274`
- **현상**: `if (hourUnknown) return ...` 가드가 isArchiveMode 체크 없이 동작. 보관함 재생 모드에서도 hourUnknown(대표 프로필 시간미상) 이면 가드에 막혀 저장된 풀이 못 봄
- **영향**: 과거 시간 알았던 시점에 받은 자미두수 기록을, 이후 프로필을 시간 미상으로 바꾸면 보관함 재생 못 함
- **수정안**: `if (hourUnknown && !isArchiveMode)` 또는 archive mode 시 record.birth_time 기준 재판정

### 11. 토정·자미두수 페이지 자체 타임아웃 vs fortuneService 타임아웃 중복
- **위치**: `fortuneService.ts:136 (AI_CLIENT_TIMEOUT_MS=55_000)` vs `ZamidusuResultPage.tsx:188~195 (45_000)` / `TojeongResultPage.tsx:123~129 (45_000)`
- **현상**: 페이지 측 45초 타임아웃이 먼저 발화 → setError + negative cache. fortuneService 의 fetch 는 진행 중이라 늦게 도착하면 abort 후 unhandled
- **영향**: 큰 부작용 없으나, 사용자가 45초에 에러 보고 재시도 누르면 negative cache 1분 걸려 즉시 같은 에러 재노출
- **수정안**: 페이지 측 타임아웃 50~55초로 정렬, 또는 발화 시 `useReportCacheStore.invalidate(...)` 호출 후 retry

### 12. TojeongIntroCard — UI 이모지
- **위치**: `src/pages/TojeongResultPage.tsx:474, 405~417`
- **현상**: `📖`, `✓`, `!` 사용. 디자인 가이드(우주/코스믹) 일관성에 어긋날 수 있음
